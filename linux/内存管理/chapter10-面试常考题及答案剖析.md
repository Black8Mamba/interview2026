## 基础概念类

### Q1: 虚拟内存解决了什么问题?

**答案要点：**

1. **扩大地址空间**
   - 进程可以使用比实际物理内存更大的虚拟地址空间
   - 32位系统可使用4GB虚拟空间，64位更大

2. **内存保护**
   - 每个进程有独立的虚拟地址空间
   - 防止进程访问非法内存或越界访问其他进程

3. **公平分配内存**
   - 操作系统统一管理物理内存
   - 按需分配给各个进程

4. **简化内存分配**
   - 应用程序看到的是连续的虚拟地址
   - 无需关心物理内存碎片化

5. **支持内存共享**
   - 共享库、共享内存等机制的基础

**延伸问题：**
- 虚拟内存的缺点是什么？
- 64位系统还需要虚拟内存吗？

### Q2: 什么是Buddy算法?

**答案要点：**

Buddy(伙伴)系统是一种物理内存分配算法：

1. **按2的幂次分配**
   - 最小单位是1页(4KB)
   - 按1, 2, 4, 8, 16...页分配

2. **伙伴关系**
   - 两个相邻的同等大小空闲块互为伙伴
   - 伙伴可以合并成更大的块

3. **分配流程**
   - 查找满足大小的最小空闲块
   - 没有则从更大的块分裂

4. **释放流程**
   - 释放时检查伙伴是否空闲
   - 空闲则合并，递归向上

**示例：**
```
分配3页 → order=2(4页)，多余1页放入order=0
释放 → 检查伙伴，合并成更大块
```

**延伸问题：**
- Buddy算法的缺点是什么？
- 如何减少内存碎片？

### Q3: 为什么需要多级页表?

**答案要点：**

1. **节省内存**
   - 一级页表需要4MB(1M条目 × 4B)
   - 多级页表按需分配，节省空间
   - 未使用的虚拟地址不需要页表

2. **层次结构**
   - 4级页表：PML4 → PDPT → PD → PT
   - 每级索引虚拟地址的不同部分

3. **TLB配合**
   - 多级页表减少内存占用
   - TLB缓存常用映射

**延伸问题：**
- ARM64和x86的页表级数有什么区别？
- 页表项包含哪些信息？

## 分配器类

### Q4: vmalloc和kmalloc的区别?

**答案要点：**

| 特性 | kmalloc | vmalloc |
|------|---------|---------|
| 虚拟地址 | 连续 | 连续 |
| 物理地址 | 连续 | 非连续 |
| 分配大小 | ~4MB | 无限制 |
| 性能 | 快 | 较慢 |
| 实现 | Buddy系统 | 页表映射 |

**适用场景：**

- kmalloc: 内核数据结构、DMA缓冲区
- vmalloc: 大块缓冲区、模块加载

**延伸问题：**
- 什么情况下必须用vmalloc？
- vmalloc的实现原理是什么？

### Q5: kmalloc和kzalloc的区别?

**答案要点：**

| 函数 | 说明 |
|------|------|
| kmalloc | 分配内存，内容未初始化 |
| kzalloc | 分配并清零，相当于kmalloc + memset(0) |

```c
// 等价于
void *p = kmalloc(size, GFP_KERNEL);
if (p)
    memset(p, 0, size);

// 就是kzalloc
void *p = kzalloc(size, GFP_KERNEL);
```

**性能对比：**
- kzalloc额外需要memset时间
- 对于大块内存，差别明显

### Q6: Slab分配器的工作原理?

**答案要点：**

1. **设计思想**
   - 缓存内核常用对象(task_struct, dentry等)
   - 避免重复初始化对象

2. **实现方式**
   - kmem_cache管理特定类型对象
   - 每个cache包含多个slab页
   - 每个slab包含多个对象

3. **三种实现**
   - Slab: 原始实现，功能全
   - **Slub**: 当前默认，简单高效
   - Slob: 嵌入式，代码量小

4. **着色(cache coloring)**
   - 减少缓存冲突
   - 优化CPU缓存命中率

**延伸问题：**
- 为什么Slub比Slab更流行？
- Slab如何提高性能？

### Q7: 页面分配标志(GFP_*)的含义?

**答案要点：**

**常用标志：**

| 标志 | 说明 | 使用场景 |
|------|------|----------|
| GFP_KERNEL | 普通分配，可阻塞 | 进程上下文 |
| GFP_ATOMIC | 原子分配，不阻塞 | 中断/软中断 |
| GFP_HIGHUSER | 高优先级用户 | 用户进程 |
| GFP_NOIO | 不进行IO | 文件系统内部 |
| GFP_NOFS | 不调用文件系统 | 块设备 |
| GFP_DMA | 从DMA区域分配 | DMA缓冲区 |
| GFP_HIGHMEM | 从高端内存分配 | 高端内存 |

**组合示例：**
```c
// 中断中分配，可能使用高端内存
GFP_ATOMIC | __GFP_HIGHMEM

// DMA分配，从DMA区域
GFP_KERNEL | GFP_DMA
```

## 机制类

### Q8: 缺页异常的处理流程?

**答案要点：**

1. **触发**
   - CPU访问未映射的虚拟地址
   - MMU产生缺页异常

2. **进入内核**
   - 保存现场
   - 调用do_page_fault()

3. **地址检查**
   - 检查地址是否合法
   - 找到对应的VMA

4. **处理缺页**
   - 调用handle_mm_fault()
   - 分配物理页面
   - 建立页表映射

5. **返回**
   - 更新页表项
   - 刷新TLB
   - 返回用户态重试指令

**分类：**
- 合法缺页：分配物理页
- 非法缺页：发送SIGSEGV

**延伸问题：**
- COW在哪里处理？
- swap如何参与？

### Q9: LRU算法原理?

**答案要点：**

1. **基本思想**
   - 最近最少使用的页面最可能被淘汰
   - 基于"过去访问模式预测未来"

2. **Linux实现**
   - 双向链表
   - 分为active/inactive两组
   - 被访问移入active，长期不访问移入inactive

3. **回收顺序**
   - 优先回收inactive
   - 进一步分为文件页和匿名页

4. **swappiness**
   - 控制文件页vs匿名页回收倾向
   - 值越大越倾向于回收匿名页

**延伸问题：**
- 为什么需要两个LRU链表？
- LRU有什么缺点？

### Q10: 什么是OOM及处理流程?

**答案要点：**

1. **触发条件**
   - 所有内存回收失败
   - 分配请求无法满足

2. **处理流程**
   - 触发OOM Killer
   - 选择一个进程杀死
   - 释放其内存

3. **选择进程**
   - 选择内存消耗最大的
   - 考虑oom_score_adj
   - 避免系统关键进程

4. **查看日志**
   ```bash
   dmesg | grep -i "out of memory"
   cat /var/log/syslog | grep OOM
   ```

**延伸问题：**
- 如何避免OOM？
- oom_score_adj如何调整？

### Q11: 如何检测内存泄漏?

**答案要点：**

1. **kmemleak**
   ```bash
   # 启用
   echo scan > /sys/kernel/debug/kmemleak

   # 查看报告
   cat /sys/kernel/debug/kmemleak
   ```

2. **/proc/meminfo观察**
   ```bash
   # 持续观察内存增长
   watch -n 1 'cat /proc/meminfo | grep -E "MemFree|MemAvailable"'
   ```

3. **内核配置**
   ```
   CONFIG_DEBUG_KMEMLEAK=y
   CONFIG_MEMCG=y
   ```

4. **用户态工具**
   - valgrind --leak-check=full
   - mtrace (glibc)

### Q13: 大页内存优缺点?

**答案要点：**

**优点：**
1. **减少TLB miss**
   - 大页减少TLB条目数
   - 提高地址转换效率

2. **减少页表内存**
   - 2MB vs 4KB，减少512倍页表

3. **提高顺序访问性能**
   - 大块顺序IO效率高

**缺点：**
1. **内存浪费**
   - 内部碎片，可能浪费内存

2. **分配困难**
   - 需要连续物理内存
   - 可能分配失败

3. **管理复杂**
   - 需要预先配置

**延伸问题：**
- THP和HugePage的区别？
- 什么场景适合大页？

### Q14: mmap原理与应用场景?

**答案要点：**

**原理：**
- 将文件或设备映射到虚拟地址空间
- 访问映射区域触发page fault
- 按需从磁盘读取

**应用场景：**

1. **文件IO**
   - 高效读写大文件
   - 内存映射文件

2. **共享内存**
   - 进程间共享数据
   - 无需copy

3. **内存分配**
   - 大块匿名映射
   - 比malloc高效

```c
// 示例：文件映射
int fd = open("file", O_RDONLY);
void *addr = mmap(NULL, 4096, PROT_READ, MAP_SHARED, fd, 0);

// 示例：匿名映射
void *addr = mmap(NULL, 4096, PROT_READ|PROT_WRITE,
                  MAP_PRIVATE|MAP_ANONYMOUS, -1, 0);
```

### Q15: 用户态内存分配方式对比?

**答案要点：**

| 方式 | 原理 | 特点 |
|------|------|------|
| malloc | brk/sbrk | 小块，默认<128KB |
| mmap | 页表映射 | 大块 |
| calloc | brk + 清零 | 分配+初始化 |
| realloc | 重新分配 | 调整大小 |

**选择建议：**
- 小块(<128KB): malloc
- 大块(>1MB): mmap
- 频繁分配释放: 内存池

## 实战类

### Q16: 实际内存问题排查案例?

**案例1：内存泄漏**

**现象：**
- 内存使用持续增长
- 可用内存逐渐减少

**排查步骤：**
```bash
# 1. 观察内存趋势
watch -n 1 'free -h'

# 2. 找到内存使用最多的进程
ps aux --sort=-rss | head

# 3. 检查/proc/PID/status
cat /proc/PID/status | grep -E "VmRSS|VmSize"

# 4. 使用kmemleak检测内核泄漏
```

**案例2：OOM**

**现象：**
- 系统变慢
- 应用被杀死
- dmesg有OOM信息

**排查步骤：**
```bash
# 1. 查看OOM日志
dmesg | tail | grep -i oom

# 2. 查看内存使用
cat /proc/meminfo

# 3. 调整oom参数
echo -17 > /proc/PID/oom_score_adj
```
