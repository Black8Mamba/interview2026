# Per-CPU - 面试题

## 基础概念

### Q1: Per-CPU 变量的优势是什么？

**答：**

1. **无锁设计**：每个 CPU 访问自己的副本，无需加锁
2. **缓存友好**：数据在 CPU 本地缓存，减少缓存争用
3. **无竞争**：CPU 之间不存在数据竞争
4. **简化代码**：无需考虑并发问题

---

### Q2: 什么场景适合使用 Per-CPU 变量？

**答：**

1. **CPU 本地统计**：如每 CPU 事件计数、网络包计数
2. **缓存友好的数据结构**：如每 CPU 缓存
3. **避免竞争的数据**：每个 CPU 独立处理的任务
4. **中断处理**：如中断统计、嵌套计数

---

### Q3: 使用 Per-CPU 变量需要注意什么？

**答：**

1. **CPU 热插拔**：CPU 上线/下线时需要特殊处理
2. **内存占用**：每个变量占用 N 份内存（N=CPU数）
3. **初始化**：Per-CPU 变量需要正确初始化
4. **跨 CPU 访问**：需要额外机制进行 CPU 间通信

---

## 进阶问题

### Q4: get_cpu_var 和 this_cpu_read 的区别？

**答：**

| 函数 | 行为 | 使用场景 |
|------|------|----------|
| `get_cpu_var(var)` | 禁止抢占 + 获取指针 | 需要长时间使用变量 |
| `this_cpu_read(var)` | 只读取当前 CPU | 快速读取 |

```c
/* get_cpu_var - 禁止抢占 */
get_cpu_var(counter)++;
/* 做其他操作 */
put_cpu_var(counter);

/* this_cpu_read - 不禁止抢占 */
this_cpu_inc(counter);  /* 原子操作，自动处理 */
```

---

### Q5: Per-CPU 与锁的性能对比？

**答：**

Per-CPU 在高并发场景下性能远优于锁：

- **锁**：获取/释放有开销，可能阻塞
- **Per-CPU**：无锁，无阻塞，缓存友好

但在以下场景使用锁更合适：
- 需要 CPU 间共享数据
- 数据量较大
- 访问模式不确定
